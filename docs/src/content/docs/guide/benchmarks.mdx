## Run Benchmarks Yourself

<Aside type="tip">
The only honest benchmark is one you run on your own hardware.
</Aside>

```bash
git clone https://github.com/egeominotti/bunqueue.git
cd bunqueue
bun install

# Run all benchmarks
bun run bench

# Individual benchmarks
bun run bench:throughput
bun run bench:latency
bun run bench:internal
```

## Throughput Benchmark

Measures jobs pushed and processed per second.

| Operation | Ops/sec | Relative |
|-----------|---------|----------|
| Single Push | **~55,000** | ████████░░░░░░░░ |
| Bulk (10) | **~95,000** | █████████████░░░ |
| Bulk (100) | **~120,000** | ██████████████░░ |
| Bulk (1000) | **~145,000** | ████████████████ |

<Aside>
Batch operations significantly improve throughput. Batch size of 100-1000 provides best performance.
</Aside>

## Latency Benchmark

Measures time to complete operations in milliseconds.

| Percentile | Push | Process | Round-trip |
|------------|------|---------|------------|
| **p50** | 0.05ms | 0.02ms | 0.08ms |
| **p95** | 0.12ms | 0.08ms | 0.20ms |
| **p99** | 0.25ms | 0.15ms | 0.40ms |
| **p99.9** | 0.60ms | 0.35ms | 0.95ms |

<Aside type="tip">
Sub-millisecond latency even at p99.9 percentile.
</Aside>

## Memory Usage

Memory footprint under different workloads.

| Queue Depth | RSS Memory | Heap Used | Per Job |
|-------------|------------|-----------|---------|
| Idle | **~50 MB** | ~25 MB | - |
| 10,000 jobs | **~65 MB** | ~40 MB | ~1.5 KB |
| 100,000 jobs | **~150 MB** | ~100 MB | ~1.0 KB |
| 1,000,000 jobs | **~400 MB** | ~300 MB | ~350 B |

<Aside>
Memory usage scales linearly. Efficient ~100 bytes per job at scale.
</Aside>

## Internal Optimizations

bunqueue uses optimized data structures internally.

```bash
bun run bench:internal
```

<Tabs>
  <TabItem label="Skip List">
    Used for priority queue ordering. O(log n) insert vs O(n) array splice.

    | Elements | Array Splice | Skip List | Speedup |
    |----------|-------------|-----------|---------|
    | 1,000 | 12ms | 2ms | **6x** |
    | 5,000 | 180ms | 8ms | **22x** |
    | 10,000 | 650ms | 15ms | **43x** |
  </TabItem>

  <TabItem label="LRU Cache">
    Doubly-linked list vs Map delete+set for move-to-front.

    | Cache Size | Naive | Optimized | Speedup |
    |------------|-------|-----------|---------|
    | 1,000 | 45ms | 12ms | **3.7x** |
    | 5,000 | 52ms | 14ms | **3.7x** |
    | 10,000 | 58ms | 15ms | **3.9x** |
  </TabItem>

  <TabItem label="MinHeap">
    Used for delayed job scheduling. O(log n) operations.

    | Elements | Sorted Array | MinHeap | Speedup |
    |----------|-------------|---------|---------|
    | 1,000 | 8ms | 1ms | **8x** |
    | 5,000 | 95ms | 5ms | **19x** |
    | 10,000 | 320ms | 10ms | **32x** |
  </TabItem>
</Tabs>

## Why bunqueue is Fast

<CardGrid>
  <Card title="Native SQLite" icon="seti:db">
    Direct FFI bindings via `bun:sqlite`. WAL mode for concurrent reads/writes. Memory-mapped I/O.
  </Card>
  <Card title="Efficient Data Structures" icon="list-format">
    Skip lists for O(log n) priority ordering. MinHeap for delayed jobs. LRU cache with O(1) access.
  </Card>
  <Card title="Batch Optimization" icon="rocket">
    Single transaction for bulk operations. Prepared statements reused. 32 shards minimize lock contention.
  </Card>
  <Card title="Zero Overhead" icon="approve-check">
    JSON stored directly in SQLite. No protocol encoding/decoding. MessagePack for binary data.
  </Card>
</CardGrid>

## Benchmark Environment

For reproducible results, note your environment:

```bash
# System info
uname -a
bun --version

# CPU info (macOS)
sysctl -n machdep.cpu.brand_string

# CPU info (Linux)
cat /proc/cpuinfo | grep "model name" | head -1
```

**Reference hardware** (used for example numbers):
- Mac Studio, Apple M1 Max, 32GB RAM
- macOS 26.2 (Tahoe)
- Bun 1.3.8
- SSD storage
