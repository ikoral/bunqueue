## Quick Start

Get bunqueue MCP running in under 60 seconds:

```bash
# 1. Install
bun add bunqueue

# 2. Add to your AI client
claude mcp add bunqueue -- bunx bunqueue-mcp

# 3. Ask your agent
# "Add a job to the emails queue with data {to: 'user@test.com', subject: 'Welcome'}"
```

That's it. Your AI agent now has access to 70 queue management tools.

### Remote Server (TCP Mode)

Connect to a running bunqueue server instead of embedded SQLite:

```json
{
  "mcpServers": {
    "bunqueue": {
      "command": "bunx",
      "args": ["bunqueue-mcp"],
      "env": {
        "BUNQUEUE_MODE": "tcp",
        "BUNQUEUE_HOST": "localhost",
        "BUNQUEUE_PORT": "6789",
        "BUNQUEUE_TOKEN": "your-auth-token"
      }
    }
  }
}
```

## Connection Modes

|                  | Embedded (default)           | TCP (remote)                     |
| ---------------- | ---------------------------- | -------------------------------- |
| **How it works** | Direct SQLite access         | Connects to bunqueue server      |
| **Setup**        | Zero config                  | Set `BUNQUEUE_MODE=tcp`          |
| **Best for**     | Local dev, single-machine    | Production, shared queues        |
| **Data**         | `./data/bunq.db`             | Remote server handles storage    |

### HTTP Handlers (3 tools)

Auto-process jobs via HTTP requests. The AI agent registers a handler on a queue, and the MCP server spawns an embedded Worker that calls the HTTP endpoint for every job.

| Tool | Description |
|------|-------------|
| `bunqueue_register_handler` | Register an HTTP handler on a queue and start auto-processing |
| `bunqueue_unregister_handler` | Remove handler and stop its worker |
| `bunqueue_list_handlers` | List all active HTTP handlers |

<Aside type="tip">
Combine with `bunqueue_add_cron` for recurring tasks. Example: register a handler on "meteo" queue, then create a cron that pushes jobs every 10 seconds. The handler auto-processes each job via HTTP.
</Aside>

**Example: "Check weather every 10 seconds"**

```bash
$ claude

> Register a handler on "meteo" queue that calls the OpenWeather API

✓ bunqueue_register_handler
  queue: "meteo"
  method: GET
  url: "https://api.openweathermap.org/data/2.5/weather?q=Milan&appid=xxx"
  Worker started.

> Create a cron that pushes a job to "meteo" every 10 seconds

✓ bunqueue_add_cron
  name: "check-meteo"
  queue: "meteo"
  schedule: "*/10 * * * * *"

# Now every 10s: cron creates job → worker calls API → result saved
# Agent can check results anytime with bunqueue_get_job_result
```

## Error Handling

All 73 tools return structured error responses with the MCP `isError: true` flag:

```json
{
  "error": "Human-readable error message"
}
```

Not-found responses for `bunqueue_get_job`, `bunqueue_get_job_by_custom_id`, `bunqueue_get_progress`, and `bunqueue_get_cron` also return `isError: true`. No raw stack traces or internal details are ever exposed.

## Processing Jobs

There are two ways to process jobs. Choose based on your use case:

### Option A: HTTP Handlers (agent-managed, no code required)

Use `bunqueue_register_handler` to let the MCP server auto-process jobs via HTTP calls. The agent registers a handler, and an embedded Worker spawns automatically. No external process needed.

```
┌────────────┐     ┌──────────────┐     ┌──────────────┐     ┌─────────────┐
│ Cron / Add │────▶│  Job Queue   │────▶│  Embedded    │────▶│  HTTP API   │
│   Job      │     │  (waiting)   │     │  Worker      │     │  endpoint   │
└────────────┘     └──────────────┘     └──────────────┘     └─────────────┘
                                              │
                                              ▼
                                        ┌──────────────┐
                                        │ Job Result   │
                                        │ (completed)  │
                                        └──────────────┘
```

**Best for:** API polling, webhooks, external service calls, monitoring, health checks.

The agent controls everything: register, schedule, monitor, unregister. No deployment needed.

### Option B: External Worker (custom logic)

For complex processing that requires custom code (database writes, file processing, business logic), deploy a Worker process:

```typescript
// worker.ts
import { Worker } from 'bunqueue/client';

new Worker('emails', async (job) => {
  const { to, subject, body } = job.data;

  await job.updateProgress(50, 'Sending email...');
  await sendEmail(to, subject, body);
  await job.log('Email sent successfully');

  return { sent: true, timestamp: Date.now() };
}, { embedded: true });
```

```bash
bun run worker.ts
```

**Best for:** Email sending, image processing, database operations, complex business logic.

The agent orchestrates (add jobs, schedule crons, monitor), the Worker executes.

## Troubleshooting

### MCP server not connecting

```bash
# Verify installation
bunx bunqueue-mcp --help

# Check if Bun is in PATH
which bun

# Test manually
echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"1.0"}}}' | bunx bunqueue-mcp
```

### Jobs stuck in waiting

Make sure a Worker is running:

```bash
bun run worker.ts
```

### Claude not seeing tools

1. Restart Claude Desktop/Code
2. Check config file syntax (valid JSON)
3. Verify path to `bunqueue-mcp` is correct
4. Run `bunx bunqueue-mcp` manually to check for errors
