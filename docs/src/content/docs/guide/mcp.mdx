## Quick Start

<Steps>

1. **Install bunqueue**

   ```bash
   bun add bunqueue
   ```

2. **Connect your AI client**

   ```bash
   claude mcp add bunqueue -- bunx bunqueue-mcp
   ```

3. **Start using it**

   Ask your agent: *"Add a job to the emails queue"* — it works immediately.

</Steps>

<Aside type="tip">
  That's all. Your agent now has access to 73 tools for queue management, scheduling, monitoring, and automated job processing.
</Aside>

### TCP Mode (Remote Server)

Connect to a running bunqueue server instead of local SQLite:

```json
{
  "mcpServers": {
    "bunqueue": {
      "command": "bunx",
      "args": ["bunqueue-mcp"],
      "env": {
        "BUNQUEUE_MODE": "tcp",
        "BUNQUEUE_HOST": "your-server.com",
        "BUNQUEUE_PORT": "6789",
        "BUNQUEUE_TOKEN": "your-auth-token"
      }
    }
  }
}
```

|                  | Embedded (default)           | TCP (remote)                     |
| ---------------- | ---------------------------- | -------------------------------- |
| **How it works** | Direct SQLite access         | Connects to bunqueue server      |
| **Setup**        | Zero config                  | Set `BUNQUEUE_MODE=tcp`          |
| **Best for**     | Local dev, single-machine    | Production, shared queues        |
| **Data**         | `./data/bunq.db`             | Remote server handles storage    |

## HTTP Handlers

The killer feature for AI agents. Register a URL on a queue, and bunqueue auto-processes every job by making HTTP calls. No external worker. No deployment. The agent controls everything.

```
┌──────────┐     ┌───────────┐     ┌────────────┐     ┌────────────┐
│ Cron or  │────▶│   Queue   │────▶│  Embedded  │────▶│  HTTP API  │
│ Add Job  │     │ (waiting) │     │  Worker    │     │  endpoint  │
└──────────┘     └───────────┘     └────────────┘     └────────────┘
                                          │
                                          ▼
                                   ┌────────────┐
                                   │ Job Result │
                                   │ (saved)    │
                                   └────────────┘
```

<CardGrid>
  <Card title="bunqueue_register_handler" icon="add-document">
    Register an HTTP handler on a queue. Spawns a worker that auto-processes jobs via HTTP requests.
  </Card>
  <Card title="bunqueue_unregister_handler" icon="close">
    Remove a handler and stop its worker.
  </Card>
  <Card title="bunqueue_list_handlers" icon="list-format">
    List all active HTTP handlers and their worker status.
  </Card>
</CardGrid>

### Example: Weather monitoring every 10 seconds

```bash
$ claude

> Register a handler on "meteo" that calls the OpenWeather API

✓ bunqueue_register_handler
  queue: "meteo"
  method: GET
  url: "https://api.openweathermap.org/data/2.5/weather?q=Milan&appid=xxx"
  Worker started. Jobs in "meteo" will be processed via GET.

> Create a cron that pushes a job every 10 seconds

✓ bunqueue_add_cron
  name: "check-meteo"
  queue: "meteo"
  repeatEvery: 10000

# Every 10s: cron → job → HTTP call → result saved
# Check anytime with bunqueue_get_job_result
```

<Aside type="tip">
  HTTP handlers support `GET`, `POST`, `PUT`, `DELETE` with custom headers, request body templates, and configurable timeouts (1-120 seconds).
</Aside>

## Processing Jobs

Two approaches depending on your use case:

<CardGrid>
  <Card title="HTTP Handlers" icon="external">
    Register a URL, bunqueue calls it for every job. No code, no deployment. Best for API calls, webhooks, monitoring.
  </Card>
  <Card title="External Worker" icon="seti:typescript">
    Deploy a Worker process with custom logic. Best for database writes, file processing, business logic.
  </Card>
</CardGrid>

### External Worker example

For processing that requires custom code:

```typescript
import { Worker } from 'bunqueue/client';

new Worker('emails', async (job) => {
  const { to, subject, body } = job.data;
  await job.updateProgress(50, 'Sending...');
  await sendEmail(to, subject, body);
  return { sent: true, timestamp: Date.now() };
}, { embedded: true });
```

The agent orchestrates (add jobs, schedule crons, monitor). The Worker executes.

## Error Handling

All 73 tools return structured error responses with `isError: true`:

```json
{
  "error": "Human-readable error message"
}
```

Not-found responses for `get_job`, `get_job_by_custom_id`, `get_progress`, and `get_cron` also return `isError: true`. No stack traces or internal details are ever exposed.

## Troubleshooting

### MCP server not connecting

```bash
# Verify installation
bunx bunqueue-mcp --help

# Check Bun is in PATH
which bun

# Test manually
echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"1.0"}}}' | bunx bunqueue-mcp
```

### Jobs stuck in waiting

No worker is processing the queue. Either register an HTTP handler or deploy a Worker:

```bash
# Option A: HTTP handler (via agent)
# Ask your agent: "Register a handler on <queue> that calls <url>"

# Option B: External worker
bun run worker.ts
```

### Claude not seeing tools

1. Restart Claude Desktop or Claude Code
2. Check config file syntax (valid JSON)
3. Verify `bunqueue-mcp` path is correct
4. Run `bunx bunqueue-mcp` manually to check for errors
